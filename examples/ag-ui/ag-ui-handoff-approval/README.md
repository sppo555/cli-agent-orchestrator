# AG-UI Handoff Approval Example

Demonstrates the `AgentHandoffWithApproval` L2 construct from the AG-UI
construct library, covering both CI-safe offline mode and live server mode.

## What it shows

- `classify_reason(provider, raw_prompt)`: deterministic, total classifier
- `on_provider_waiting()`: creating interrupt records with UI emission
- `resume(interrupt_id, decision)`: lock-guarded exactly-once resolution
- Idempotent re-resume (first resolution wins)
- `expire(terminal_id)`: zero-keystroke expiration on state transition
- `projection()`: pending/total view of the interrupt registry
- `RecordingUiEmitter` capturing `approval_card` intents

## Running

### Offline mode (CI-safe, no server)

```sh
./examples/ag-ui/ag-ui-handoff-approval/run.sh
```

Exercises the construct API directly with synthetic data. No credentials,
no server, no network calls.

### Live mode (server + scripted prompts)

```sh
CAO_APPROVAL_LIVE=1 ./examples/ag-ui/ag-ui-handoff-approval/run.sh
```

Starts `cao-server` with `CAO_MOCK_CLI_SCRIPTED_PROMPTS=1`, waits for a
terminal to enter WAITING_USER_ANSWER, then resolves the resulting interrupt
via the REST endpoint.

## Live real-provider procedure (approval-mode claude_code)

For a full human-in-the-loop demo with a real Claude Code provider:

1. **Configure the profile** - Create or select a claude_code profile in your
   CAO config that does NOT use `--dangerously-skip-permissions` (the default
   approval mode).

2. **Start CAO** with AG-UI enabled:
   ```sh
   export CAO_AGUI_ENABLED=1
   uv run cao-server
   ```

3. **Launch a task** that will trigger a permission prompt:
   ```sh
   cao launch --provider claude_code --yolo \
       "Create a file called /tmp/hello.txt with the word hello"
   ```

4. **Observe the interrupt** - The terminal enters WAITING_USER_ANSWER.
   The ApprovalBridge picks it up and calls `on_provider_waiting`, which:
   - Classifies the prompt (e.g., `claude-code:permission_request`)
   - Creates an Interrupt record
   - Emits an `approval_card` generative-UI frame on the AG-UI stream

5. **Resolve via REST**:
   ```sh
   # List pending interrupts
   curl http://localhost:9889/agui/v1/approvals | jq .pending

   # Approve the first one
   INTERRUPT_ID=$(curl -s http://localhost:9889/agui/v1/approvals | jq -r '.pending[0].id')
   curl -X POST "http://localhost:9889/agui/v1/approvals/${INTERRUPT_ID}/resume" \
       -H 'content-type: application/json' \
       -d '{"decision": "approve"}'
   ```

6. **Or via the AG-UI run plane** (stock client):
   ```sh
   curl -X POST http://localhost:9889/agui/v1/run \
       -H 'content-type: application/json' \
       -d '{
         "threadId": "t1",
         "runId": "r1",
         "resume": [{"interruptId": "...", "status": "resolved", "payload": {"approved": true}}]
       }'
   ```

7. **Observe resolution** - The construct translates the decision to the
   provider-specific keystroke (Enter for claude_code approve) and delivers
   it to the terminal. A second `approval_card` frame is emitted with
   `resolved: true`.

## Architecture

```
Terminal (WAITING_USER_ANSWER)
    |
    v
ApprovalBridge (event-bus consumer)
    |
    v
AgentHandoffWithApproval.on_provider_waiting()
    |  -> classify_reason()
    |  -> Interrupt record
    |  -> emit("approval_card", {...})
    v
REST: POST /agui/v1/approvals/{id}/resume
    |
    v
AgentHandoffWithApproval.resume(id, decision)
    |  -> _translate_decision(provider, decision)
    |  -> AnswerDelivery.send_input() or .send_special_key()
    v
Terminal resumes
```
