## 1. Executive summary

The remediation correctly closes the two code-level findings from the first review:

1. A declared CAO terminal no longer becomes a global operator when terminal metadata
   lookup fails.
2. Legacy `scope="global"` plus `memory_type="project"` wiki entries are filtered from
   normal recall, related-memory expansion, and dynamically generated terminal memory
   context.

The new audit and quarantine commands also provide a reasonable explicit cleanup path for
the runtime memory store.

However, a separate persistent injection path remains open. CAO memory plugins write
context into provider-native instruction files such as `AGENTS.md` and
`.claude/CLAUDE.md`. Existing polluted blocks in those files can be loaded by Codex or
Claude before the post-create plugin refresh runs, and an empty newly generated context
does not remove the old block.

This is not theoretical: the repository's current `AGENTS.md` still contains the known
polluted global entry `slice-a3-e2e-tts-sentinel`.

Because the remediation report claims that legacy pollution cannot enter prompt
injection, this remaining path is a blocker.

## 3. First-review findings that are now closed

### 3.1 Terminal identity lookup now fails closed

Affected implementation:

```text
src/cli_agent_orchestrator/mcp_server/server.py
```

The remediation adds `MemoryTerminalContextError` and distinguishes two caller modes:

- No `CAO_TERMINAL_ID`: intentional operator invocation; `None` context remains valid.
- `CAO_TERMINAL_ID` present: terminal metadata must be verified or the memory operation is
  refused.

The following cases now fail closed:

- Transport timeout.
- Connection refusal.
- HTTP failure.
- Non-object metadata.
- Terminal ID mismatch.
- Missing or empty required metadata fields.

A working-directory lookup failure preserves the verified
`caller_scope="project"`. Project writes then fail because a project ID cannot be resolved,
while global writes remain denied. The caller is not promoted to operator.

**Status:** Closed.

### 3.2 Legacy global/project wiki entries are filtered from normal reads

Affected implementation:

```text
src/cli_agent_orchestrator/services/memory_service.py
```

The remediation introduces a shared `_is_invalid_global_project()` policy and applies it
inside `_parse_wiki_file()`. Because this parser is shared by memory materialization, the
guard covers:

- Normal metadata recall.
- BM25 and hybrid recall result materialization.
- Related-memory expansion.
- `get_memory_context_for_terminal()` dynamic context construction.

Valid global `user`, `feedback`, and `reference` memories remain eligible.

**Status:** Closed for wiki-backed read and dynamic context paths.

### 3.3 Audit and quarantine operations are explicit

The remediation adds:

```bash
cao memory scope-audit
cao memory quarantine-global-project KEY
cao memory quarantine-global-project KEY --apply
```

Positive properties include:

- Audit is read-only.
- Quarantine is dry-run by default.
- Applying quarantine preserves a copy before removing live persistence.
- No project identity is guessed.
- Reports avoid returning memory body content.

**Status:** Acceptable for the reviewed scope.

## 4. Remaining blocking finding

### BLOCKER — Stale provider instruction files bypass the new read filter

Affected files:

```text
src/cli_agent_orchestrator/services/terminal_service.py
src/cli_agent_orchestrator/plugins/builtin/codex_memory.py
src/cli_agent_orchestrator/plugins/builtin/claude_code_memory.py
src/cli_agent_orchestrator/plugins/builtin/kiro_cli_memory.py
```

CAO has two relevant memory delivery mechanisms:

1. Dynamic memory context generated by `MemoryService`.
2. Provider-native instruction files written by memory plugins.

The new parser guard protects the first mechanism. It does not automatically clean blocks
that were written into the second mechanism before this remediation.

### 4.1 Provider initialization happens before memory plugin refresh

In `terminal_service.py`, CAO creates and initializes the provider before dispatching the
`post_create_terminal` plugin event.

The synchronous sequence is:

```text
Create provider
    ↓
provider.initialize()
    ↓
Codex or Claude reads AGENTS.md / CLAUDE.md
    ↓
Build Terminal object
    ↓
dispatch post_create_terminal
    ↓
memory plugin refreshes the managed block
```

Relevant locations:

```text
src/cli_agent_orchestrator/services/terminal_service.py:310-344
src/cli_agent_orchestrator/services/terminal_service.py:376-385
```

Therefore, a stale block can already be included in the current model session before the
plugin replaces the file.

The deferred initialization path also does not establish a safe ordering guarantee. It
schedules provider initialization in the background before dispatching the plugin event,
creating a race between provider startup and instruction-file refresh.

### 4.2 Empty context does not remove an existing managed block

Each provider memory plugin returns early when the newly generated context is empty:

```text
codex_memory.py:84-89
claude_code_memory.py:78-83
kiro_cli_memory.py:74-79
```

Consider a project whose only available memory is the legacy polluted global/project
entry:

```text
Legacy polluted block already exists in provider instruction file
    ↓
New parser correctly filters the invalid runtime memory
    ↓
Generated context is empty
    ↓
Plugin returns without touching the instruction file
    ↓
Old polluted block remains indefinitely
```

The existing plugin tests explicitly require an empty context not to create or modify the
target file. They do not distinguish between:

- No target file exists, where doing nothing is correct.
- A target file contains an old CAO-managed block, where the block must be removed.

### 4.3 Current workspace provides direct evidence

The repository's current untracked `AGENTS.md` begins with a CAO-managed block containing:

```text
- [project] grok-phase0-evidence-outcome: ...
- [global] slice-a3-e2e-tts-sentinel: ...
```

The second entry is the known EngVocabulary-specific memory that was incorrectly stored
globally. Its presence proves that provider instruction files can preserve the polluted
content independently of the runtime memory parser.

Claude Code uses the same architecture with `.claude/CLAUDE.md`, which is directly
relevant to the reported `b7c205bb` supervisor incident.

### 4.4 Impact

Without fixing this path:

- A new Codex or Claude terminal can ingest stale cross-project context during startup.
- Quarantining the runtime memory does not necessarily remove previously injected copies.
- A project with no remaining valid memory may never trigger replacement of its stale
  managed block.
- The remediation report's claim that legacy pollution cannot enter prompt injection is
  false for provider-native instruction files.

## 5. Required remediation

### 5.1 Refresh before provider initialization

CAO must refresh or scrub the provider-native CAO memory block before the provider starts
and reads its instruction files.

Acceptable designs include:

- Introduce a pre-provider-initialization plugin phase.
- Perform memory-file preparation before calling `provider.initialize()`.
- Split provider construction from process launch and run the memory refresh between
  those phases.

The implementation must provide deterministic ordering for both synchronous and deferred
initialization.

### 5.2 Remove stale managed blocks when context is empty

If a target file already exists and contains CAO markers, an empty current context must
remove the CAO-managed block while preserving all user-authored content.

Required behavior:

| Current context | Target state | Required action |
|---|---|---|
| Empty | Target absent | Do nothing; create nothing |
| Empty | Target exists without CAO block | Do nothing |
| Empty | Target contains CAO block | Remove only managed block |
| Non-empty | Target absent | Create managed block |
| Non-empty | Target contains old block | Atomically replace managed block |

Kiro's dedicated `.kiro/steering/cao-memory.md` file may be deleted when current context is
empty, since the entire file is CAO-managed. Codex and Claude must preserve surrounding
user-authored content.

### 5.3 Add end-to-end regression tests

Tests should cover all three providers and prove:

1. Seed a stale instruction file containing a legacy global/project entry.
2. Seed the runtime store with the same invalid entry.
3. Create a new terminal for a different project.
4. Ensure the instruction file is scrubbed before provider initialization.
5. Capture the provider's first loaded context or startup command environment.
6. Assert the invalid entry is absent from the first model turn.
7. Assert valid project and global context remains present where applicable.
8. Assert user-authored instruction-file content is byte-preserved outside the managed
   block.

Unit tests for `_strip_existing_block()` alone are insufficient because the principal bug
is lifecycle ordering.

### 5.4 Extend maintenance visibility

The audit or cleanup workflow should identify provider-native files containing stale
CAO-managed blocks, or provide a safe scrub command for known project paths. Runtime memory
quarantine alone does not remove these derivative copies.

## 7. Updated acceptance criteria

The remediation may be approved when all of the following hold:

1. A declared terminal identity cannot fail open into operator authority.
2. New global/project writes fail before persistence.
3. Legacy global/project runtime entries cannot enter recall or dynamic context.
4. Existing provider-native managed blocks are refreshed before provider startup.
5. Empty current context removes stale managed blocks without creating new files.
6. User-authored `AGENTS.md` and `CLAUDE.md` content is preserved.
7. Kiro's stale dedicated memory steering file is removed safely.
8. Synchronous and deferred terminal initialization have the same deterministic ordering.
9. First-turn integration tests prove that stale content is absent from the actual model
   context, not only from `MemoryService` return values.
10. Runtime quarantine and provider-file cleanup responsibilities are documented.

## 8. Final verdict

`d433ba8` successfully closes the first review's fail-open terminal identity bug and the
runtime wiki read-path gap. It does not yet close the provider-native instruction-file
path, which can preserve and load the same cross-project content before the cleanup plugin
runs.

**Verdict: `CHANGES REQUESTED`**
