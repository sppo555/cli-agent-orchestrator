## 1. 背景（給修繕者快速對齊）

### 1.1 症狀

1. Codex 已完成一輪（輸出含 review artifact、`APPROVED`、`Worked for …`、回傳 supervisor 的 `send_message`）。
2. CAO 仍回報 terminal 狀態 `processing`。
3. 新 inbox 訊息停在 pending，不會進入 Codex。
4. 人工開啟 Web terminal（attach / resize / SIGWINCH）後：
   - Codex 重畫 idle prompt
   - 狀態轉 `COMPLETED` / `IDLE`
   - pending message 才真正送入

### 1.2 根因

與 4.13 init 修復相同：**無人觀看的 tmux TUI 不會把最終 idle frame 完整刷進 `pipe-pane`**。

差異在生命週期：

| 階段 | 舊修復覆蓋？ | 結果 |
|------|--------------|------|
| `provider.initialize()` | 是（headless viewer + periodic SIGWINCH） | init 期間可 unstick |
| 延續使用既有 terminal，一輪結束後 | 否 | status 卡 `PROCESSING`，inbox 不配送 |

CAO inbox 只在 receiver 為 `IDLE` / `COMPLETED` 時配送。StatusMonitor（尤其 Codex 依賴底部 idle `›` / pyte screen）收不到乾淨 idle frame 時，狀態不會翻轉。

### 1.3 現有修復做了什麼（`2f3c678`）

| 檔案 | 變更 |
|------|------|
| `services/render_viewer.py` | 抽出 `nudge_once()`；新增 `nudge_unattended_render()` 一次 attach + redraw + cleanup |
| `services/terminal_service.py` | 新增 `nudge_terminal_render(terminal_id)`；event-driven backend 跳過 |
| `services/inbox_service.py` | reconciliation 在 stale pending + 仍 `PROCESSING` 時觸發 redraw；per-terminal 60s cooldown |
| tests | inbox / render_viewer 單元測覆蓋主路徑 |

**正確決策（請保留）：**

- 不開 inbox 旁路；仍走 StatusMonitor → status event → `deliver_pending`。
- 不向 agent 送文字 / Enter / Escape / Ctrl+C。
- 只在 grace 過期的 pending 上觸發。
- tmux/pipe-pane only；herdr 不動。

## 3. 必做：E2E 驗收（A1）

### 3.1 目標

證明：**無人開 Web terminal** 時，continued Codex terminal 完成後，pending message 可被自動配送。

### 3.2 步驟

1. 以本 worktree build 啟動獨立測試 CAO server（勿直接動生產）。
2. 建立 / 延續一個 **Codex** worker terminal（模擬 reviewer 多輪，不是只測冷啟動 init）。
3. 送出一輪會跑數十秒以上的任務，讓其真正進入 `PROCESSING` 再結束。
4. **全程不要**打開該 terminal 的 Web viewer。
5. 任務結束後立即（或結束後）對該 terminal `send_message` 一則可辨識 sentinel（例如 `E2E-NUDGE-PROBE-<ts>`）。
6. 觀察時序：

| 檢查點 | 預期 |
|--------|------|
| 任務結束當下 status | 可能短暫或持續 `processing`（此即 bug 條件） |
| message 狀態 | 先 `PENDING` |
| 約 ≥ `INBOX_RECONCILE_GRACE_SECONDS`（預設 30s）後 | reconciliation 觸發 redraw |
| redraw 後 | StatusMonitor 轉 `idle` / `completed` |
| 其後 | message 變 `DELIVERED`，Codex 實際收到 sentinel |
| 全程 | 無人工 attach Web terminal |

### 3.3 失敗判定

任一下列即 E2E 失敗，必須做 **A2** 後重測：

- 60s 內 status 仍 `processing` 且 message 仍 pending。
- 只有人工開 Web terminal 後 message 才送入。
- redraw 造成 agent 收到非預期輸入 / 中斷 turn（不應發生）。
- herdr / event-driven backend 被誤觸發 viewer（不應發生）。

### 3.4 建議記錄

E2E 結果請補回本目錄或 commit message，至少包含：

- CAO server 版本 / commit
- terminal id
- status 時間線（processing → idle/completed）
- pending → delivered 時間
- 是否觸發 nudge（見 A4 log）

## 4. P1：加強 redraw 可靠度（A2）

### 4.1 問題

`nudge_unattended_render()` 目前：

```text
start() → nudge_once(shrink=True) → sleep(0.25s) → stop()
```

但歷史 `bb98aea` 已證明：

- 被動 attach **不夠**
- init 需要 **週期 SIGWINCH**（每 2.5s）才能穩定翻轉 status

0.25s one-shot 對 Codex 重畫可能偏緊；若 client 過早 detach，idle frame 可能尚未進入 `pipe-pane`。失敗後要再等 **60s cooldown**。

### 4.2 建議實作（擇一，優先 #1）

**方案 1（推薦）：雙 nudge + 還原尺寸**

```text
start(periodic=False)
nudge_once(shrink=True)
sleep(0.3–0.5)
nudge_once(shrink=False)   # 還原 rows，淨尺寸不變，第二次 SIGWINCH
sleep(settle)              # settle 建議預設 1.0s
stop()
```

**方案 2：拉長 settle + 維持 viewer**

```text
start(periodic=False)
nudge_once(shrink=True)
sleep(1.0–1.5)             # 保持 attach 期間讓 repaint 流入 pipe-pane
nudge_once(shrink=False)   # 可選還原
stop()
```

**方案 3：短生命週期 periodic（最接近 init）**

```text
start(periodic=True)
sleep(3.0)                 # 允許 _nudge_loop 至少跑 1 次（loop 首等 2.5s）
stop()
```

注意：方案 3 若沿用現有 `_nudge_loop`，前 2.5s 不會 nudge；需配合立即 `nudge_once()` 或縮短 first wait。

### 4.3 修改位置

- `src/cli_agent_orchestrator/services/render_viewer.py`
  - `nudge_unattended_render()`
  - 可選：`_RenderViewer.start(periodic_nudge: bool = True)`
- 對應測試：`test/services/test_render_viewer.py`
  - 斷言 nudge 次數、stop 必呼叫、settle 參數

### 4.4 驗收

- 單元測綠
- A1 E2E 在 **不開 Web terminal** 下通過
- 真 busy terminal（有 pending）被 resize 時，agent 不中斷、不吞輸入

## 5. P2：Cooldown 與可觀測性（A3 / A4）

### 5.1 Cooldown 寫入時機（A3）

現況（`inbox_service.py`）：

```python
if now - last >= _REDRAW_COOLDOWN_SECONDS:
    self._last_redraw_nudge[terminal_id] = now  # 成功前就寫
    terminal_service.nudge_terminal_render(terminal_id)
```

**問題：** attach 失敗也燒滿 60s。

**建議：**

```python
if now - last >= _REDRAW_COOLDOWN_SECONDS:
    ok = terminal_service.nudge_terminal_render(terminal_id)
    if ok:
        self._last_redraw_nudge[terminal_id] = now
    else:
        # 可選：短失敗 cooldown，避免緊迴圈打 tmux
        self._last_redraw_nudge[terminal_id] = now - _REDRAW_COOLDOWN_SECONDS + _REDRAW_FAIL_COOLDOWN
```

或更簡單：失敗不寫 cooldown，但在 `nudge_terminal_render` 內對 attach 失敗做 rate-limit。

### 5.2 Logging（A4）

請在 recovery 路徑加可搜尋 log（避免只靠 debug）：

| 事件 | 等級 | 建議訊息欄位 |
|------|------|----------------|
| 決定 nudge | INFO | `terminal_id`, `status`, `reason=stale_pending_processing` |
| nudge 成功 | INFO | `terminal_id`, `session`, `window` |
| nudge 失敗 | WARNING | `terminal_id`, `reason`（no metadata / event-backend / attach fail） |
| cooldown 跳過 | DEBUG | `terminal_id`, `remaining_s` |

方便對照 b5121378 類問題時直接 `rg "unattended render|redraw nudge"`。

## 6. P3：程式整潔（A5–A8）

### 6.1 分離 init / recovery 生命週期（A5）

`nudge_unattended_render()` 呼叫完整 `start()`，會啟動 `_nudge_loop`（首等 2.5s），one-shot 存活期內通常跑不到，多餘。

```python
def start(self, *, periodic_nudge: bool = True) -> bool:
    ...
    if periodic_nudge:
        self._nudge = threading.Thread(target=self._nudge_loop, daemon=True)
        self._nudge.start()
```

recovery：`viewer.start(periodic_nudge=False)`。

### 6.2 更新模組 docstring（A6）

`render_viewer.py` 頂部仍寫：

> during an active task the CLI emits output continuously, so no viewer is needed after init

此句已被 b5121378 否證。請改成：

- init：長駐 viewer + periodic SIGWINCH
- post-init / continued：inbox reconciliation 可做 one-shot（或 short-lived）redraw recovery
- 兩者皆不送 agent input；僅 tmux backend

### 6.3 常數化（A7）

建議移到 `constants.py`（或與 `INBOX_RECONCILE_*` 並列）：

| 常數 | 建議預設 | 說明 |
|------|----------|------|
| `INBOX_REDRAW_COOLDOWN_SECONDS` | `60` | per-terminal redraw 冷卻 |
| `INBOX_REDRAW_SETTLE_SECONDS` | `1.0`（若採 A2） | attach 後等待 repaint |
| `INBOX_REDRAW_FAIL_COOLDOWN_SECONDS` | `15`（可選） | 失敗重試冷卻 |

硬編碼 `_REDRAW_COOLDOWN_SECONDS = 60.0` 可刪。

### 6.4 測試補齊（A8）

| 測試 | 期望 |
|------|------|
| cooldown 過期後第二次 reconcile 會再 nudge | mock `time.monotonic` |
| ready（IDLE/COMPLETED）不 nudge | 已有 COMPLETED；可加 IDLE |
| event-driven backend 不 nudge | 已有 |
| `nudge_unattended_render` 失敗時 stop 仍呼叫 | start False / nudge False |
| 若實作雙 nudge | 斷言 `nudge_once` call count 與 shrink 順序 |
| 若改 cooldown-on-success | 失敗後立即第二次 reconcile 仍可 nudge |

- `test/services/test_inbox_service.py` — `TestReconcileOrphanedMessages`
- `test/services/test_render_viewer.py`

## 7. 不要改的邊界（除非 E2E 證明需要）

1. **不要**在 `PROCESSING` 時強制 `deliver_pending` 旁路送訊（會打到 busy agent）。
2. **不要**為了這題只加 Codex regex 而忽略通用 tmux redraw（其他 Ink/TUI provider 同根因）。
3. **不要**預設擴大 nudge 狀態到 `WAITING_USER_ANSWER`（可能干擾真 interactive prompt）。若線上出現「卡在非 PROCESSING 且 redraw 可解」再開 issue。
4. **不要**在 redraw 路徑送任何 keystrokes。
5. **不要**修改真實 CAO session 的 `window-size` policy；繼續只 pin **viewer grouped session**（`8666514` / 現況 `set-option -t viewer_session`）。
