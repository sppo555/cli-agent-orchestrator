本修復延伸 4.13 原有的 headless render viewer。原本 4.13 只處理 worker 初始化時，無人觀看的 tmux TUI 沒有刷新，導致狀態卡住；本次補上「延續使用既有 terminal，一輪工作結束後沒有輸出最終 idle frame」的恢復路徑。

## 2. 使用者觀察到的問題

Terminal `b5121378` 的 Codex reviewer 已完成工作，輸出中已存在：

- 最終 review artifact；
- `APPROVED` verdict；
- `Worked for 2m 44s` 完成標記；
- 回傳 supervisor 的 `send_message` 紀錄。

但 CAO 仍回報 terminal 狀態為 `processing`。當新訊息送往該 terminal 時，訊息無法立即進入 Codex；必須人工開啟 Web terminal，訊息才真正送入。

## 3. 根因判定

CAO 的 inbox 正常配送條件是 receiver 處於 `IDLE` 或 `COMPLETED`。若狀態仍是 `PROCESSING`，訊息會保留在 pending，避免把新任務混入尚未完成的 turn。

無人觀看的 tmux TUI 在一輪工作完成後，可能沒有把最後的 idle/completed 畫面完整刷新到 `pipe-pane`。因此：

1. Codex 實際已完成；
2. StatusMonitor 沒收到可辨識的最終畫面；
3. CAO 保留舊的 `PROCESSING` 狀態；
4. Inbox reconciliation 呼叫正常 delivery gate，但因狀態不是 ready 而跳過；
5. 人工開啟 terminal 會造成 viewer attach/resize/SIGWINCH；
6. TUI 重新繪製目前畫面，StatusMonitor 才辨識為 ready；
7. pending message 隨後被正常配送。

這與 4.13 原始修復的根因相同，但生命週期不同：原始修復只在 `provider.initialize()` 期間維持 render viewer，初始化完成後即停止。

## 4. 修復設計

### 4.1 觸發條件

既有 inbox reconciliation 找到超過 grace period 的 pending message 後：

1. 先走原本的 `deliver_pending()`；
2. 若 terminal 仍被判定為 `PROCESSING`；
3. 且該 terminal 距離上次成功 redraw 已超過 60 秒（失敗則 15 秒後重試）；
4. 執行一次無輸入的 terminal redraw recovery。

### 4.2 Redraw 動作

Redraw recovery：

- 建立短暫的 grouped tmux viewer session；
- attach 到指定 terminal window；
- 將 viewer session 的 `window-size` 設為 `manual`；
- recovery viewer 不啟動 init 使用的 periodic nudge thread；
- 先縮小一列觸發第一次 SIGWINCH；
- 等候 0.4 秒後還原尺寸，觸發第二次 SIGWINCH；
- 保持 attach 1.0 秒，讓 repaint 穩定流入 `pipe-pane`；
- 無論成功或失敗都停止 viewer 並清除 grouped session。

此流程不會向 Codex、Claude 或其他 agent 傳送文字、Enter、Escape、Ctrl+C 等輸入。

### 4.3 安全限制

- 只有 tmux/pipe-pane backend 會執行 redraw。
- event-driven backend 直接跳過。
- 成功 redraw 使用 60 秒 cooldown；失敗使用 15 秒短 cooldown。
- pending message 必須已超過既有 reconciliation grace period。
- 真正仍在執行的 agent 最多收到一次 TUI resize/repaint，不會收到新訊息強制注入。
- 訊息仍由既有 StatusMonitor → ready event → InboxService 路徑配送，不另開旁路。

## 7. 已知限制與風險

### 7.1 尚未執行真實 session E2E

單元與 API 回歸測試已通過，但尚未用安裝此 worktree build 的 CAO server 重現以下完整流程：

1. Codex terminal 完成工作但保持無人觀看；
2. CAO 狀態錯誤停留在 `processing`；
3. 建立 pending inbox message；
4. reconciliation 自動 redraw；
5. 狀態轉為 ready；
6. 訊息在沒有人工開啟 terminal 的情況下送入。

合併前建議將此流程列為必要驗收。

### 7.2 Cooldown 是記憶體內狀態

CAO server 重啟後 cooldown 會重置。這不影響正確性；最壞情況是 server 重啟後對仍卡住的 terminal 再執行一次 redraw。

### 7.3 長時間執行中的 terminal

若 agent 真正執行超過 reconciliation grace period，而且同時有 pending message，系統可能每 60 秒做一次 redraw。這不會提交輸入，但會產生 resize/SIGWINCH。若真實負載顯示 TUI 對 resize 敏感，可把 cooldown 提高，或再加入 output-quiescence timestamp gate。

### 7.4 未修改 Codex provider status pattern

本次處理的是「最終畫面沒有流入 StatusMonitor」的通用 tmux 問題，而不是只針對 Codex 完成字樣增加 regex。這可同時涵蓋其他 TUI provider，但若 repaint 後 Codex 仍誤判，就需另行補充 Codex completed fixture/status detection。
